[PRD] 판매종료·스토어 제거 상품의 UP 상태 자동 해제

영역라이브 스튜디오 · 스토어판매 / UP PM이건무 VOC마녀옷장 셀러 (CX 서연 접수) JIRAGRIPPGM-3292 ↗ JIRAGRIPSVR-10068 ↗ 상태DEV 배포 완료 · QA 6/11~6/15 서버민욱 QA병성 (1.5MD)
개요 용어 문제 정의 해결 방안·범위 User Flow 기능 명세 핵심 정책 영향 범위 QA 계획 일정·히스토리 오픈 이슈

1. 개요 (Overview)

배경

그립 스튜디오의 스토어판매 방송에서 셀러는 최대 5개 상품을 UP 하여 방송 화면(시청자 뷰)에 노출할 수 있습니다. 그런데 현재는 UP 상태인 상품을 판매종료(판종)하거나 스토어·방송에서 제거해도 서버의 UP 상태가 해제되지 않아, 화면에서 사라진 상품이 UP 슬롯을 계속 점유합니다. 이 때문에 셀러는 화면상 UP 상품이 5개 미만인데도 "UP은 최대 5개 까지만 가능합니다" 알림에 막혀 신규 상품을 UP 할 수 없습니다.

VOC 원문 (마녀옷장 셀러 → CX 서연):
  • "UP 상태로 판매를 종료(또는 스튜디오에서 제거)한 상품까지 UP 수량에 카운팅됨"
  • "최대 5개까지 UP 가능한데, 판종한 2개 상품 때문에 타 브랜드 상품 중 3개만 UP 가능"
  • 증상 화면: 7개 상품만 스토어에 불러왔는데 "UP 5개만 가능" 얼럿 발생 — 이전에 판매종료 + 스토어 OFF 처리한 상품이 UP 상태로 남아있던 것이 원인
  • 제안: 품절 및 스토어에서 뺀 상품은 자동으로 UP이 풀리도록 변경 요청

목적

  • UP 슬롯 정합성 확보 — 방송에서 제거된 상품의 UP 상태를 자동 해제하여, 서버 카운트와 셀러 화면이 일치하도록 한다
  • 셀러 운영 차질 제거 — 라이브 중 신규 상품 UP이 막히는 상황(매출 직결)을 방지한다
  • 모든 인입 경로 일관성 — 어드민/비즈센터/앱/스튜디오 등 어느 경로로 제거해도 동일하게 동작

기대 효과

  • 동일 유형 VOC·CS 재발 차단 (셀러가 원인을 인지할 수 없는 유형의 버그)
  • 라이브 중 UP 운영 신뢰도 회복 — UP은 방송 매출에 직접 영향을 주는 핵심 기능

2. 용어 정의

용어정의
UP스토어판매/한정판매 방송에서 상품을 방송 화면(시청자 뷰)에 카드로 노출하는 기능. 방송당 최대 5개
UP 슬롯UP 가능한 5개의 자리. 서버가 UP 상태 상품 수를 기준으로 검증
판종 (판매종료)셀러/운영자가 상품 판매를 종료 처리하는 것
스토어 OFF상품을 스토어(방송 판매 목록)에서 내리는 것
유령 UP (본 문서 한정)방송 화면·상품 목록에서는 사라졌지만 서버에는 UP 상태로 남아 슬롯을 점유 중인 상품
스토어판매 / 한정판매그립 라이브의 두 가지 판매 방송 유형. UP 기능은 양쪽 모두 존재하며, 동기화 스케줄러 구성이 다름

3. 문제 정의 (AS-IS)

재현 시나리오 — 좌측 프로토타입 AS-IS 모드에서 직접 재현 가능

순서행동결과
1셀러가 상품 7개를 스토어 방송에 불러옴판매상품 7개 노출
2과거 방송에서 UP 상태로 판종 + 스토어 OFF 처리한 상품 2개 존재서버 UP 카운트 = 2 (화면에는 보이지 않음)
3신규 상품 UP 시도3개까지만 성공
44번째 UP 시도"UP은 최대 5개 까지만 가능합니다" 알림 — 화면상 UP은 3개뿐

근본 원인

  • 방송에서 상품을 제거할 때 UP 상태 해제 로직이 없음 — UP 카운트 모수에 제거·판종된 상품이 계속 포함됨
  • 최초 VOC는 어드민에서 제거하는 케이스였으나, 서버 조사 결과 어드민·비즈센터·앱을 통한 상품 제거 모두 동일한 버그로 확인됨 (서버 민욱)
  • UP 카드 데이터는 syncActivatedProductsInfo 스케줄러 → liveService.syncStoreProductUpList()에서 동기화되고 있으나, 제거 시점의 UP 해제는 다루지 않음
셀러 관점의 심각성: 셀러는 "판종한 상품이 슬롯을 점유한다"는 사실을 알 수 없습니다. 화면에는 UP 3개만 보이는데 최대 알림이 뜨므로 시스템 오류로 인지하게 되고, 라이브 진행 중 UP을 활용한 판매 운영이 막힙니다.

4. 해결 방안 & 범위 (TO-BE)

상품이 스토어(방송)에서 제거되는 시점에 해당 상품의 UP 상태를 자동 해제하고 UP 카운트에서 차감합니다. 인입 경로(어드민/비즈센터/앱/스튜디오)와 무관하게 동일하게 동작해야 합니다.

구분항목비고
In Scope스토어에서 뺀(방송에서 제거한) 상품의 UP 자동 해제 — 상품 추가/제거 모든 엔드포인트 대응스토어판매 · 한정판매 모두
Out of Scope품절 시 UP 자동 해제VOC 원문에는 포함되었으나 이번 개발 범위에서 제외, 후속 검토 (서버 민욱 공유)
Out of Scope스토어 방송 UP 동기화 스케줄러 주기 단축한정판매 라이브 대상 단주기 스케줄러(범수 작업)의 처리 속도 모니터링 후 스토어 포함 여부 별도 결정
좌측 프로토타입 TO-BE 모드: 상품 카드의 ⋯ 메뉴에서 어떤 경로(스튜디오/어드민/비즈센터/앱/판종)로 제거해도 UP이 즉시 자동 해제되고 슬롯이 반환되는 것을 확인할 수 있습니다.

5. User Flow

핵심 분기:
  • UP 상태 상품 제거 → UP 해제 + 카운트 차감 + 방송 화면 카드 동기화
  • 비(非) UP 상품 제거 → 기존과 동일, 영향 없음
  • AS-IS(빨간 점선) → 해제 로직 없이 제거만 처리되어 유령 UP 발생

6. 기능 요구사항 (Functional Requirements)

좌측 프로토타입에서 PRD 모드가 켜진 상태입니다. 프로토타입의 번호 뱃지를 클릭하면 해당 명세 카드가 강조되고, 반대로 아래 카드 제목을 클릭하면 프로토타입의 해당 요소가 하이라이트됩니다.

1
UP 버튼 & 최대 5개 제한영역: 스튜디오(PC) · 서버  |  위치: 스토어판매 > 상품 카드 우측

현행 동작

  • UP 버튼 클릭 → 서버가 UP 상태 상품 수를 검증 → 5개 초과 시 "UP은 최대 5개 까지만 가능합니다" 알림
  • 문제: 검증 모수에 판종·제거된 상품이 포함되어 화면과 카운트가 불일치

변경 사항

  • UP 카운트 모수에서 방송에서 제거된 상품이 자동으로 빠지도록 보장 (기능명세 3의 결과)
  • UP 버튼 자체의 UI/동작 변경 없음 — 검증 결과만 정합성 확보
2
상품 제거 동선 — 모든 엔드포인트영역: 어드민 · 비즈센터 · 앱 · 스튜디오  |  프로토타입: 상품 카드 ⋯ 메뉴

기능 구분

방송 판매 목록에서 상품이 빠지는 모든 인입 경로가 대상입니다. 최초 VOC는 어드민 케이스였으나 조사 결과 모든 경로에서 동일 버그가 확인되었습니다.

상세 동작

  • 어드민 — 운영자가 방송 상품 제거
  • 비즈센터 — 셀러가 상품 관리에서 제거/스토어 OFF
  • 앱 — 셀러 앱에서 방송 상품 제거
  • 스튜디오 — PC 스튜디오에서 방송 상품 제거
  • 판종(판매종료)으로 인해 스토어에서 빠지는 케이스 포함

개발 고려사항

  • 상품 추가/제거 관련 모든 엔드포인트 전수 조사 필요 (서버) — 일부 경로 누락 시 동일 VOC 재발
3
UP 자동 해제 처리 ★ 핵심영역: 서버  |  프로토타입: 상단 "서버 UP 슬롯" 바

기능 구분

이번 과제의 핵심. 상품이 방송에서 제거되는 시점에 서버가 해당 상품의 UP 상태를 해제하고 카운트를 차감합니다.

상세 동작

  • 제거 트랜잭션 내에서 UP 상태 확인 → UP이면 해제 처리
  • 해제 즉시 UP 슬롯 반환 — 신규 상품 UP 가능
  • 스토어판매·한정판매 방송 모두 동일 적용

정책

  • 해제는 비가역 — 제거된 상품을 다시 방송에 추가해도 UP 상태는 복원되지 않음 (셀러가 재 UP)
  • 품절로 인한 자동 해제는 이번 범위 제외 (후속 검토)
4
자동 해제 안내 (제안)영역: 스튜디오(PC)  |  프로토타입: 안내 문구 · 토스트

기능 구분

서버 단독 처리라 별도 UI 스펙은 없으나, 셀러 인지를 위해 프로토타입에서 두 가지를 제안합니다. (선택 사항 — 서버 작업과 분리 가능)

제안 상세

  • 스토어판매 안내 문구에 자동 해제 조건 명시: "판매종료 또는 품절 시 UP 상태가 자동 해제됩니다" — 실제 동작과 문구 일치 필요 (이번 범위는 제거 시 해제까지)
  • UP 상품이 제거되어 자동 해제될 때 스튜디오 토스트로 고지 (TO-BE 모드에서 확인 가능)
5
UP 상품만 보기 — 목록 정합성영역: 스튜디오(PC)  |  위치: 판매상품 목록 우측 토글

상세 동작

  • 토글 ON 시 UP 상태 상품만 필터링 노출
  • UP 자동 해제가 반영된 후 필터 결과와 서버 카운트가 항상 일치해야 함
  • QA 확인 포인트: 해제 직후 필터 목록에서 해당 상품이 정상적으로 제외되는지
6
방송 화면 상품 카드 동기화영역: 서버 · 시청자 앱  |  프로토타입: 하단 "방송 화면 미리보기"

기능 구분

UP 로직 변경에 따라 방송 화면(시청자 뷰)의 UP 상품 카드 동기화에 회귀가 없는지 확인이 필요합니다.

상세 동작

  • 스토어 방송: syncActivatedProductsInfo 스케줄러 내 liveService.syncStoreProductUpList()에서 UP 카드 동기화
  • 한정판매 라이브: UP 카드 데이터 동기화 주기를 줄인 별도 스케줄러 운영 중 (범수 작업)
  • UP 자동 해제 시 시청자 방송 화면에서 해당 상품 카드가 정상 제거/갱신되는지 확인

개발 고려사항

  • 단주기 스케줄러의 처리 속도 모니터링 후, 스토어 방송 포함 여부를 별도 논의 (현재 미지원)

7. 핵심 정책 (Policy)

항목정책
UP 카운트 모수현재 방송(스토어)에 노출 중인 상품 가운데 UP 상태인 상품만 포함. 제거·판종된 상품은 제외
해제 트리거스토어(방송)에서 상품 제거 — 인입 경로(어드민/비즈센터/앱/스튜디오) 무관 동일 동작
품절이번 범위 제외. VOC 원문에는 포함된 요청이므로 후속 과제로 관리
재추가 시 UP 복원복원하지 않음 — 셀러가 필요 시 다시 UP (단순성·예측 가능성 우선)
기존 유령 UP 데이터배포 시점에 이미 존재하는 유령 UP의 정리 방안(일괄 배치 vs 스케줄러 자연 정리) 서버 확인 필요 — 오픈 이슈

8. 영향 범위 (Impact)
