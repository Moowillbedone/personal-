1. 배경 및 목적 (Background & Objectives)

배경

Grip 서비스의 로그인 채널은 현재 카카오(62.53%) 단일 의존도가 매우 높은 구조이며, 그 뒤로 email(17.20%, 셀러 인입 추정) · facebook(8.20%) · id_pw(6.30%) · apple(5.77%, iOS only) 순으로 분포합니다. 이메일 가입은 그리퍼(셀러) 인입이 포함되어 있어 순수 소비자 기준으로 보면 카카오 편중이 더 심각한 상황입니다. 경쟁 커머스(배민·컬리 등)는 이미 네이버 간편로그인을 주요 채널로 운영 중이거나 도입 예정으로, 네이버 생태계 유저 확보를 위한 도입이 필요한 시점입니다.

목적

  • 가입 채널 확대로 카카오 편중을 완화하고 로그인 실패/피로도 감소
  • 네이버 생태계 유저 유입으로 신규 가입 전환 모수 확대
  • 운영 효율화 — 이메일 가입 경로에 섞여 있는 소비자 인입을 별도 SNS로 분리

기대 효과

  • 첫 로그인 화면에서의 이탈률 개선 (특히 비(非)카카오 유저)
  • 중장기적으로 Google 간편가입 도입 시 Android 비중 대응 기반 마련

정성적 목표인 "가입 전환 기대" 를 정량적으로 측정하기 위해 아래 지표를 추적합니다.

3. IA 구조

  • P1 로그인 페이지 — 카카오/네이버 2-top 버튼 + 기타 아이콘
  • P2 네이버 인증 — 네이버 OAuth 2.0 로그인 (외부 네이버 페이지)
  • P3 동의 화면 — 개인정보 제 3자 제공 동의 (필수/선택 분리)
  • P4 Grip 회원가입 — 네이버 Prefill + 추가 입력 + 약관 동의
  • P5 홈 — 가입 완료 직후 랜딩
  • P6 로그인 관리 — 마이페이지 > 설정 내 연결 계정 CRUD

4. User Flow

핵심 분기:
  • 기존 회원(naver 연결) → P3·P4 스킵, P5 홈 직행
  • 이메일 충돌(他 SNS 선가입) → 계정 연결 제안 모달 → 연결 시 홈, 취소 시 로그인
  • 신규 → P3(동의) → P4(가입) → P5(홈)

5. 기능 요구사항 (Functional Requirements)

좌측 프로토타입에서 PRD 모드가 활성화된 상태이며, 각 번호 뱃지를 탭하면 해당 항목이 강조됩니다. 반대로 우측 번호 카드를 탭하면 프로토타입의 해당 요소가 하이라이트됩니다.

4

기능 구분

네이버 로그인 페이지에서 사용자가 이미 네이버 세션을 보유한 경우 계정 선택 UI 노출. 미로그인 상태면 ID/PW 입력 화면.

상세 동작

  • 네이버 세션 O → 가입된 ID 리스트 노출 + "다른 아이디로 로그인" 옵션
  • 네이버 세션 X → ID/PW 입력 폼
  • 선택/로그인 완료 → OAuth 2.0 Authorization Code 발급 → Grip 서버 콜백

정책

  • 이 화면은 네이버 소유 UI — Grip 에서 수정 불가
  • 네이버 공식 UI 가이드를 따르며, Grip 은 "state" 파라미터로 CSRF 방지 처리만 담당

개발 고려사항

  • Authorization Code → Access Token 교환은 서버 사이드에서 처리 (Client Secret 노출 방지)
  • Token 을 사용해 네이버 프로필 API 호출 → 이름/이메일/옵션 정보 수신
  • redirect_uri 화이트리스트 관리 (Dev/Stg/Prod)

기능 구분

사용자가 네이버 아이디를 선택(탭)했을 때만 활성화되는 확정 버튼. 탭 시 네이버에 OAuth Authorization Code 발급을 요청하고, 콜백을 통해 Grip 서버로 리다이렉트됩니다.

상세 동작

  • 초기 상태: disabled · 회색 배경 (#D1D5DB) · 클릭 불가
  • 활성 조건: 네이버 계정 리스트에서 1개 선택 시 즉시 활성 (#03C75A 배경)
  • 탭 동작:
    • 기존 Grip 회원(naver 연결)이면 동의 화면(P3) 스킵 → 즉시 홈(P5)
    • 신규면 동의 화면(P3) 진입
  • 로딩 상태: 버튼 내부에 스피너 노출 + 버튼 중복 탭 방지 (debounce)

정책

  • 버튼은 선택 전까지 반드시 비활성 — 사용자 실수 클릭 방지
  • 선택 후 다른 아이디로 재선택하면 선택값이 갱신되며 버튼은 활성 유지
  • 네이버 OAuth 요청에는 CSRF 방지를 위한 state 파라미터 필수
  • redirect_uri 는 환경별(Dev/Stg/Prod) 다르게 등록된 화이트리스트 사용

개발 고려사항

  • Access Token 교환은 서버 사이드에서 처리 (Client Secret 노출 방지)
  • 네이버 응답 지연 시 최대 10초 타임아웃 → 실패 시 "잠시 후 다시 시도해주세요" 에러 토스트
  • 이벤트 로깅: naver_login_attempt { selected_idx: N }

엣지 케이스

  • 네이버 응답 실패(5xx) → 로딩 해제 + 에러 토스트 + 재시도 가능
  • 네트워크 오프라인 → "인터넷 연결을 확인해주세요" 토스트
  • 동시에 여러 번 탭 → debounce 로 첫 요청만 처리, 이후 무시

6. 핵심 정책 (Critical Policies)

6.1 기존 회원 재로그인 감지

정책: 네이버 로그인 성공 후 Grip 서버가 naverSub(고유 식별자) 또는 이메일로 user 조회 → 매칭되면 P3 동의 화면 및 P4 가입 화면을 건너뛰고 즉시 로그인.
이유: 이미 동의한 유저에게 재동의를 요구하면 경험 저해. 단, 네이버에서 수집 항목이 변경된 경우는 재동의 요청 필수 (법적 요건).

6.2 계정 충돌 처리 (이메일 중복)

시나리오: 유저 A가 과거 카카오로 foo@naver.com 이메일로 가입. 이후 같은 이메일 네이버로 간편가입 시도.
정책:
  • P3 동의 완료 후 서버에서 이메일 중복 탐지 → 계정 연결 제안 모달 노출
  • "이미 카카오로 가입된 이메일이에요. 네이버 계정을 연결할까요?" + [연결하기] / [취소]
  • [연결하기] → 기존 카카오 유저의 linked_accounts 에 'naver' 추가 → 자동 로그인
  • [취소] → 로그인 페이지 복귀

6.3 동의 데이터 재사용

  • 네이버 선택 동의 상태는 user_consent 테이블에 개별 저장 (consent_naver_nickname, consent_naver_gender, ...)
  • 추후 해당 필드를 추가로 활용해야 할 때 재동의 요청 플로우 필요

6.4 로그아웃 / 탈퇴 시 네이버 연동

  • 로그아웃: 네이버 토큰은 그대로 유지 (다음 로그인 UX 개선)
  • 탈퇴: 네이버 Unlink API 호출 + 네이버에서 받은 개인정보 즉시 파기 (개인정보보호법)

6.5 데이터 연동 (Data/Analytics)
